# PMO 项目归类规则 - 调研事业部

## 一、核心概念

### 1.1 Session 与 Issue 的关系

```
1 个本地 Session = 1 或 n 个 Linear Issue
```

- 每个 Issue 的 description 必须包含 `sessionId: <session_id>`
- 同一 Session 可能产出多个 Issue（如：先做技术调研再做竞品分析）

### 1.2 事业部信息

| 字段 | 值 |
|------|-----|
| Team ID | `0851ce16-e2d0-411f-adfe-aacc26cdab7b` |
| Key | R |
| 说明 | 深度调研与情报分析 |

## 二、Project = 调研类型

调研事业部的 Project 按调研类型划分：
- `技术调研` - 框架选型、方案评估、能力评测
- `市场调研` - 用户痛点、竞品分析、市场规模
- `投资调研` - 行业研究、公司分析、财报解读
- `舆情分析` - 社交媒体情绪、热点追踪
- `人物/公司` - 背景调查、深度扒析

**Issue = 一次完整调研**，如：
- `[技术] Remotion vs Motion Canvas`
- `[市场] AI 英语学习 App 竞品分析`
- `[投资] NVIDIA 2026 Q1 财报解读`

## 三、Labels

### 3.1 调研阶段

| Label | 说明 |
|-------|------|
| 选题 | 确定调研方向和范围 |
| 采集 | 多源信息收集 |
| 分析 | 深度分析与交叉验证 |
| 报告 | 结构化输出 |
| 应用 | 调研结论落地（推动其他 BU 决策） |

### 3.2 信息来源（可选）

| Label | 说明 |
|-------|------|
| Reddit | 基于 Reddit 数据 |
| Twitter/X | 基于 X 平台数据 |
| 小红书 | 基于小红书数据 |
| Perplexity | AI 搜索引擎 |
| 一手数据 | 直接采集的原始数据 |

**可多选**：一个 Issue 可以同时有 `分析` + `Reddit` + `Twitter/X`

## 四、Issue 创建规范

### 4.1 必填字段

```graphql
issueCreate(input: {
  title: "任务标题",
  teamId: "0851ce16-e2d0-411f-adfe-aacc26cdab7b",
  projectId: "项目ID",
  description: "sessionId: xxx\n\n任务描述",
  labelIds: ["label1", "label2"]
})
```

### 4.2 Description 模板（轻量摘要）

```markdown
sessionId: <session_id>

## 摘要
<一句话调研结论>

## Done Criteria
- [ ] 报告产出
- [ ] 可执行建议 ≥1 条

## 调研范围
- 信息源: <使用了哪些渠道>
- 关键词: <搜索关键词列表>
```

### 4.3 Document: 调研报告

通过 `doc-create --issue R-xx` 创建，详细内容独立存放：

```markdown
## 核心发现
1. **事实**: <已验证的事实，附来源>
2. **观点**: <值得关注的观点，标注来源>
3. **数据**: <关键数据点>

## 结论与建议
- 结论: ...
- 建议: ...
- 风险: ...

## 产出物
| 文件 | 说明 |
|------|------|
| `file:///path/to/report.md` | 主报告 |
| `file:///path/to/data.csv` | 结构化数据 |

## 来源链接
- ...
```

## 五、触发与归类

### 5.1 触发时机

PMO Classify Hook 在以下操作时自动触发：
- `research` skill 调用 → 调研事业部
- `research-by-reddit` skill 调用 → 调研事业部
- 大量 `api-fetch` 调用（信息采集模式）→ 调研事业部

### 5.2 归类流程

1. **匹配 Project**：根据调研内容匹配调研类型
2. **选择 Labels**：根据当前阶段和信息源打 Label
3. **创建 Issue**：提交到调研事业部 Team

### 5.3 Project 匹配规则

**匹配逻辑**：
1. 从 Session 上下文提取关键词（用户消息、搜索查询、调研报告内容）
2. 与已有 Project 的 keywords 匹配
3. 计算匹配度：高（>0.8）、中（0.5-0.8）、低（<0.5）

**匹配结果处理**：

| 匹配度 | 行为 |
|--------|------|
| 高 | 自动关联，静默处理 |
| 中 | 用 AskUserQuestion 确认 |
| 低/无 | 默认归入对应调研类型或让用户选择 |

## 七、跨 BU 协同

调研事业部是其他三个 BU 的"智囊团"，调研成果应推动决策：

| 调研类型 | 协同 BU | 场景 |
|----------|---------|------|
| 技术调研 | 产品 | 技术选型决策 → 产品 BU 采用 |
| 市场调研 | 产品/内容 | 用户痛点 → 产品方向 / 选题灵感 |
| 投资调研 | 投资 | 研究报告 → 投资决策 |
| 舆情分析 | 内容 | 热点追踪 → 内容选题 |
| 人物/公司 | 投资/内容 | 背景调查 → 投资判断 / 合作决策 |

**关联方式**：调研 Issue 完成后，如果产生了可执行建议，在目标 BU 创建关联 Issue。

## 八、Session 总结模板

Session 结束时，通过 `doc-create --issue R-xx` 创建 Document（而非追加到 description）：

**Document 标题**：`Session 总结: <调研主题>`

```markdown
## 调研概要
- 主题: ...
- 信息源: ...
- 数据量: <搜索了多少条/分析了多少篇>

## 关键发现
| 序号 | 发现 | 类型 | 可信度 | 来源 |
|------|------|------|--------|------|
| 1 | ... | 事实/观点 | 高/中/低 | ... |

## 方法论
- 搜索策略: <关键词组合>
- 分析方法: <定性/定量/混合>
- 验证方式: <交叉验证/专家验证/数据验证>

## 产出物
| 类型 | 文件 | 说明 |
|------|------|------|
| 报告 | `file://...` | 主报告 |
| 数据 | `file://...` | 原始数据 |

## 经验沉淀
- 有效: <这个搜索策略/分析方法效果好>
- 改进: <下次可以怎么做得更好>
- 知识更新: <是否需要更新 Obsidian 笔记>
```

## 九、注意事项

- 不要过于频繁打断用户，连续操作时只在关键节点询问
- 如果用户明确说"跳过"或"不分类"，尊重选择
- 新 Project 需要用户确认后才在 Linear 创建
- Labels 可多选，反映实际工作内容
- 调研报告中必须区分事实与观点
- 所有结论必须附带来源链接，不引用无法追溯的信息
